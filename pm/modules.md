# 业务能力模块化方案（Business Capability Modules）

> **决策依据**：[ADR-0004：业务能力按域模块化](./decisions/0004-business-capability-modules.md)  
> 本文档负责**随迭代更新**：域划分、包命名、首批落地顺序与模板组合关系。评审重大变更请走 `pm/rfcs/` → ADR。

## 1. 目标

将**市场常见小程序能力**按**业务域**拆成可版本化的 workspace 包（`@applet/*`），在各端（微信 / H5 / 支付宝 / 抖音等）项目中通过**依赖 + 少量胶水代码**快速拼装，避免每个项目从零复制粘贴。

**不与「脚手架」对立**：`create-applet` 负责**生成壳工程与选模板**；**模板变薄**，核心是**勾选或默认引入哪些 `@applet/*` 模块**。

## 2. 分层

| 层级           | 职责                           | 示例包 / 产物                                        |
| -------------- | ------------------------------ | ---------------------------------------------------- |
| **基础层**     | 与具体业务场景弱相关的横切能力 | `@applet/request`、`@applet/auth`、`@applet/tracker` |
| **业务域模块** | 可独立售卖/复用的垂直能力      | 支付唤起、优惠券、订单列表、会员积分入口等           |
| **模板**       | 模块组合 + 默认路由 / 占位页   | `templates/ecommerce`、`basic-react`                 |
| **CLI**        | 参数化生成、写入依赖与配置     | `create-applet` 扩展 `--modules` 等（演进）          |

依赖规则（v1 约定，细则以 RFC 为准）：

- **业务域模块** → 只允许依赖 **基础层** 与其他业务模块的 **公开 API**（禁止穿透内部实现）。
- **基础层** → 不依赖具体业务域模块。
- 跨端差异 **封装在模块内部**；模块对外暴露稳定 TypeScript 接口。

## 3. 业务域划分（初始清单，可增删）

下列域用于**规划对齐**；是否每一个都单独成包，由体积与复用度决定（可合并域，不可「巨型万能包」）。

| 域代码       | 说明        | 典型能力（关键词）           |
| ------------ | ----------- | ---------------------------- |
| `account`    | 账号 / 会员 | 登录、手机号、会员、积分入口 |
| `commerce`   | 交易 / 订单 | 购物车、下单、订单状态       |
| `payment`    | 支付        | 各端支付唤起、结果轮询       |
| `marketing`  | 营销        | 优惠券、活动、分享参数       |
| `content`    | 内容        | 富文本、媒体列表、详情壳     |
| `lbs`        | 门店 / 定位 | 定位授权、附近门店、导航     |
| `support`    | 客服 / 反馈 | 客服会话入口、工单跳转       |
| `compliance` | 合规 / 隐私 | 协议同意、埋点 consent       |

> 注：`payment` 可与 `commerce` 合并为同一包，直到拆包收益明确。

## 4. 包命名约定（草案）

- Scope 统一：`@applet/`。
- 建议：`@applet/<domain>-<short-name>`，例如 `@applet/commerce-order`、`@applet/marketing-coupon`。
- **基础层** 延续简短名：`@applet/request`、`@applet/auth`。

正式命名表在首个业务包开 PR 前，在 **RFC** 中锁一行「包名 ↔ 域 ↔ 对外 API 清单」。

## 5. 与里程碑的对应关系（简表）

| 里程碑 | 与模块化的关系                                                                               |
| ------ | -------------------------------------------------------------------------------------------- |
| M1     | `basic-react` 壳工程需预留 `packages/*` 依赖方式，避免业务全写死在 template 内               |
| M2     | **基础层**（request / auth）优先，作为所有业务模块的底座                                     |
| M5     | 业务模板 = **预组模块合** + 示例页；ecommerce 模板应对齐 commerce / payment / marketing 等域 |
