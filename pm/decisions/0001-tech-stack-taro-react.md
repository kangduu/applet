# ADR-0001：技术栈锁定 Taro + React + TypeScript

- **状态**：Accepted
- **日期**：2026-04-30
- **决策者**：项目发起人
- **关联 RFC**：—（项目启动期直接决策）
- **取代 / 被取代**：—

## 背景与上下文

项目目标是构建一个可生成 11 个端的小程序模板与脚手架。市场主流跨端方案有 Taro、uni-app、Mpx 三大派系，分别基于 React/Vue 生态、Vue 生态、增强版小程序原生语法。需要在项目启动阶段锁定一条主路线，避免维护多套并行体系导致进度失控。

约束条件：

- 团队对 React 生态更熟悉，TypeScript 经验充足
- 需要支持微信、H5 为最高优先级，并具备扩展到支付宝、抖音等端的能力
- 希望生态活跃、社区资源丰富、能复用 React 生态的组件和工具链

## 决策

**v1.x 主线锁定 Taro 4.x + React 18 + TypeScript 5。**

- Vue3 / uni-app 路线推迟到 v2.0 再开第二轨。
- CLI 预留 `--framework` 选项，但 v1.x 内只实现 React 分支。
- 鸿蒙 / RN 端通过 Taro 自身的多端能力输出，列入 P3。

## 备选方案

| 方案                       | 优势                                       | 劣势                              | 是否采用 |
| -------------------------- | ------------------------------------------ | --------------------------------- | -------- |
| Taro + React + TS          | 生态最活跃、TS 一流支持、平台覆盖最广      | 编译时方案部分平台仍依赖 webpack | ✅ 采用  |
| uni-app（Vue3）            | DCloud 商业生态、App 端 uts 能力强         | Vue3 路线与团队既有积累不匹配     | ❌       |
| Mpx                        | 性能接近原生，运行时开销最小               | 开发体验偏向小程序原生，扩展性受限 | ❌       |
| Taro + Vue3                | 同框架双语法                               | 两套模板维护成本翻倍              | ❌       |

## 后果

### 正面
- 团队学习曲线最低，能尽快进入交付节奏
- React 生态可复用（Zustand、React Query、各类 hooks）
- Taro 4 已转向 Vite，构建速度受益

### 负面 / 代价
- Vue 用户需要等到 v2.0
- 部分平台插件仍依赖 Webpack，存在双轨期（见 R-003）

### 中性
- 需要为后续 Vue 路线预留 CLI flag 与共享包的语言无关性

## 影响范围

- 代码：所有模板、CLI、共享包默认按 React + TS 实现
- 流程：CI 中加入 `tsc --noEmit` 与 React 相关 Lint
- 文档：README 第三章、第八章
