# ADR-0004：业务能力按业务域模块化（workspace 包 + 薄模板）

- **状态**：Accepted
- **日期**：2026-05-06
- **决策者**：项目维护者（与发起人目标对齐）
- **关联 RFC**：—（细则可由后续 RFC 补充 semver、依赖图）
- **取代 / 被取代**：—

## 背景与上下文

项目长期目标不仅是「生成多端工程」，还要让**市场常见小程序能力**在多个小程序产品中**快速复用**。若能力仅散落在各 `templates/*` 中复制粘贴，会导致分叉多、升级难、各端差异无法统一收敛。

需要将**按业务拆分的模块**沉淀为可版本化的 **monorepo 内 `@applet/*` 包**，模板与 CLI 负责**组合与生成**，而非承载全部业务实现。

## 决策

1. **业务能力以「业务域」为主轴线**拆分为多个 workspace 包（示例域：账号/会员、交易/订单、支付、营销、内容、LBS、客服、合规等），详见 `pm/modules.md` 维护的清单。
2. **基础横切能力**（如 HTTP、登录态、埋点）作为**底层包**优先实现（与 M2 的 request / auth 一致）；业务域包仅依赖底层与其他包的**公开 API**。
3. **业务模板**（如 ecommerce）定义为：**预选模块组合 + 默认路由/占位页 + 配置钩子**；模板代码保持**薄**，避免与具体模块实现耦合过深。
4. **CLI**（`create-applet`）演进方向包括：可选模块列表、写入 `package.json` 依赖与最小胶水代码（实现排期见 backlog，不在本 ADR 规定具体参数名）。
5. **域划分、包命名、首批落地顺序**以 `pm/modules.md` 为运行清单；重大调整需 RFC/评审并同步 `pm/changelog.md`。

## 备选方案

| 方案                        | 优势               | 劣势                         | 结论     |
| --------------------------- | ------------------ | ---------------------------- | -------- |
| 仅厚模板，无独立包          | 实现快             | 无法复用、升级成本极高       | 不采用   |
| 单包 `@applet/all` 聚合     | 引用简单           | 包体积与 tree-shaking 压力大 | 不采用   |
| 按域分包 + 薄模板（本决策） | 可版本化、组合灵活 | 需约束依赖与文档             | **采用** |

## 后果

### 正面

- 新产品可「选型 + 拉包」快速起量，而不是 fork 模板。
- 各端差异收敛在模块内部，对业务侧接口稳定。

### 负面 / 代价

- 需额外治理：**模块边界、semver、循环依赖**（见 `risks.md` R-008）。
- M2/M5 工作量需显式包含「模块化」而不仅是「写一个模板」。

### 中性

- 与现有 Taro + React + pnpm monorepo（ADR-0001/0002）一致，不引入第二技术栈。

## 影响范围

- 代码：`packages/*` 规划、未来 `templates/*` 与 `create-applet` 选型逻辑。
- 流程：`pm/modules.md` 持续维护；模块相关 breaking change 建议走 RFC。
- 文档：根 `README.md` 产品目标可在后续迭代与 §一、§四 对齐。
