# ADR-0003：v1.0 平台范围 = weapp / h5 / alipay / tt

- **状态**：Accepted
- **日期**：2026-04-30
- **决策者**：项目发起人
- **关联 RFC**：—
- **取代 / 被取代**：—

## 背景与上下文

README 列出了 11 个潜在目标平台。一次性交付全部平台会让 v1.0 周期失控，且许多平台属于长尾需求。需要明确 v1.0 的最小可行集合。

## 决策

**v1.0 必须覆盖 4 个端**（README 表中 P0）：

| 平台         | 标识     | 选入理由                         |
| ------------ | -------- | -------------------------------- |
| 微信小程序   | `weapp`  | 国内体量最大，主战场             |
| H5           | `h5`     | 浏览器调试与分享必备             |
| 支付宝小程序 | `alipay` | 金融 / 政务场景，企业客户高频    |
| 抖音小程序   | `tt`     | 字节系流量入口，电商内容场景刚需 |

后续版本扩展节奏：

- **v1.1**：百度（swan）、QQ
- **v1.2**：京东、钉钉、飞书
- **v2.x**：快手、RN、鸿蒙

## 备选方案

| 方案                     | 优势                         | 劣势                        | 是否采用 |
| ------------------------ | ---------------------------- | --------------------------- | -------- |
| 仅 weapp + h5            | 最快出 v1.0                  | 客户场景覆盖不足            | ❌       |
| weapp + h5 + alipay + tt | 覆盖主流主体场景，工程量可控 | 比最小集多约 1 周工作量     | ✅       |
| 一次到位（11 端）        | 完整                         | 至少多出 6~8 周，风险不可控 | ❌       |

## 后果

### 正面

- 一次发布即可服务多数主流业务场景
- 在 M4 阶段就能验证跨端抽象设计是否经得起 4 端考验

### 负面 / 代价

- M4 工期拉长 0.5~1 周
- 支付宝、抖音端的 UI 一致性测试需要额外资源（见 R-004）

### 中性

- 为 v1.1/v1.2 扩展打好抽象层基础，降低后续端的接入成本

## 影响范围

- 代码：CLI 平台枚举、模板的 `config/`、`project.*.json`、CI 矩阵
- 流程：CI 增加 4 端构建验证步骤
- 文档：README 第二章
