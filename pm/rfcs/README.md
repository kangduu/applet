> RFC 用于在做决策**之前**广泛收集意见。任何超过 1 周工作量、跨多个包、影响公共契约的变更都建议先开 RFC。

## 流程

1. 复制 `pm/templates/rfc.md` 到 `pm/rfcs/NNNN-kebab-case-title.md`，编号取索引最大值 +1。
2. 初始状态 `Draft`，作者完成自填后改为 `In Review`，并在团队渠道广播。
3. 评审周期建议 **3~5 个工作日**。期间通过 PR 评论 / 离线讨论收集意见，作者持续修订。
4. **通过**：状态改为 `Accepted`，同步创建对应 ADR（在 `decisions/`），把 ADR 链接回填到 RFC 顶部，并把条目追加到上方索引。
5. **拒绝**：状态改为 `Rejected`，但**保留文件**，写明拒绝理由，便于后人查阅。
6. **被新提案取代**：状态改为 `Superseded`，链接到新 RFC。

## 何时需要 RFC？

需要：

- 新增 / 变更 CLI 公共参数与交互流程
- 新增 / 调整共享包的对外 API
- 影响多个模板的工程化基线变更
- 跨多包的目录结构调整
- 引入新的核心依赖（构建器、状态管理、UI 库等）

不需要（直接 PR 即可）：

- 单个模板内部的实现重构
- Bugfix
- 文档优化
- 单元测试补充
